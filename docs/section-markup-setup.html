FeedbackAuthored in PreTeXt

Section 2.2 Setup and Randomization

The setup block defines all of the dynamically generated mathematical objects. It also specifies particular constraints on any variables in the system. The tag has an attribute seed which is used to provide a default seed for the initial state of the problem. To support WeBWorK's approach, this should be set equal to a string containing a positive integer. The block consists of a sequence of instruction elements.

Instruction elements can define parameters, variables, and expression terms. Parameters can be generated randomly or calculated in terms of previously defined parameters. Expression terms can be computed in terms of previously defined parameters or expressions using the defined variables. All of these elements have @name attribute. Parameter and expression names should start with a letter and can contain letters A-Za-z, digits 0-9, and the underscore. I have to check whether variable names have the same flexibility, but they might only allow letters.

Mustache Substitution.

Implementations of dynamic exercises will reference the parameter and expression names in different ways. For example, WeBWorK and WebAssign both use Perl syntax and so the names will be transcoded with a dollar sign $ before the name. Other implementations will just use the names directly. To retain a simple substitution schema when defining a value or an expression, we adopt mustache substitution. This means that the name of the parameter or expression needs to be surrounded by double curly braces. Variables themselves are not enclosed by braces. For example, if we had previously defined a parameter c and wanted to introduce an expression \(cx+3\text{,}\) the mustache-substitution expression would be written {{c}}*x+3.

When referring to parameters or expressions for display in PreTeXt content or to refer to expressions as objects for evaluation (as in when checking answers), we instead use XML tags and use the #var element with @name="expression-name" to match.

de-parameter

Used to define a numeric value, generated either randomly or statically.

Attributes:

@name

Name starts with letter and can contain A-Za-z, 0-9 and underscore.

@random

Value is a string specifying how the value is randomly assigned.

random='discrete' means the value will be chosen from an finite sequence of equally spaced values. Uses the additional attributes @min, @max, @by, and @nonzero.

random='uniform' means the value will be chosen uniformly between a minimum and maximum value. Uses the additional attributes @min and @max.

Need to think about markup for choosing one or more values from an arbitrary list.

@min

Value is a number for the minimum of either discrete or uniform random values. Default: min="0"

@max

Value is a number for the maximum of either discrete or uniform random values. Default: max="1". For the discrete sequence, the maximum is technically just an upper bound. Sequence values above @max are not selected

@by

Value is a number for the step size increment of a discrete sequence. Default: by="1"

@nonzero

Value is “yes” or “no”, to indicate for a discrete random choice whether the value of zero should always be excluded. Default: nonzero="no"

Content: When the parameter is not randomly assigned, the content of the element is used to define the value of the parameter. The parameter can be calculated in terms of previously defined parameters using mustache substitution. Variables and names of expressions can not be used. For randomly generated values, the content is empty.

de-variable

Define a symbol that will be treated as a variable. The default variable of \(x\) is not required to be defined.

I propose that there is no content and the name of the variable is defined with the attribute @name. I am wondering if child elements might be included to define context information about the domain where the variable is defined and values where the variable is evaluated when comparing expressions.

WeBWorK distinguishes variables as being real- or complex-valued. I do not believe WebAssign allows for complex-variable expressions. Should we have an attribute for this?

An alternative implementation would be to have a tag de-variables and then have the content be a list of the variables that are used. We would need to specify the delimiter (space or comma default?).

de-term

Used to define an expression capable of being used in comparisons for evaluation.

In my mind, I am thinking of this like a MathObject formula in WeBWorK or as an expression that is going to appear inside of a problem statement or solution. WeBWorK MathObjects and my proposed Javascript implementation allow complex manipulation of expressions, including substitution, evaluation, and differentiation. WebAssign, on the other hand, seems to think of expressions as static formulas. My proposed markup is intended to allow for manipulation. A future conversion to WebAssign would either be more complex (simple substitution should be possible using Perl functions) or require warning about unsupported features.

Attributes:

@name

Name starts with letter and can contain A-Za-z, 0-9 and underscore.

@context

Value is a string specifying how the expression is defined.

context='formula' means the content is a mustache-substitution formula.

context='substitution' means the formula is defined by a variable substitution from another formula. The content contains child elements: #start containing either a de-term defining the initial expression or #var matching a previously-defined expression; #match containing the variable being replaced; and #replace containing another expression defined by #de-term or #var.

I'm wondering about whether it would make more sense for #start and #replace to have a name attribute to match previous formulas and content only if needing to define on the spot, and similarly for #match to only have a name matching the variable and no content.

I can see adding contexts for the derivative of a previously-defined expression. I can also see this allowing non-formula contexts as well, similar to other MathObject contexts (intervals, sets, lists).
